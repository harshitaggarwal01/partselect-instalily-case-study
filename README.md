# PartSelect Case Study – Harshit Aggarwal

A PartSelect‑branded support and shopping assistant for refrigerators and dishwashers.  
It diagnoses common issues, recommends concrete parts, walks users through installation, and lets them collect parts into a cart before handing off to the real PartSelect website to complete checkout.

---

## Features

- **Domain‑bounded support**
  - Handles four intents:
    - Install instructions
    - Compatibility checks
    - Troubleshooting
    - Product information
  - Refuses out‑of‑scope questions.

- **RAG over repair guides**
  - Curated `install_guides.jsonl` and `troubleshooting_guides.jsonl`.
  - Offline embedding into `vector_index.json` with multi‑chunk guides and rich metadata.
  - Cosine similarity + metadata boosting + haiku‑based reranking for higher‑quality retrieval.

- **Dummy authentication**
  - Login and signup with simple token‑based auth.
  - Pre‑seeded demo accounts:
    - `instalily` / `instalily`
    - `harshit` / `harshit`

- **Persistent multi‑thread chat**
  - Multiple named conversations per user.
  - Threads and messages stored in `threads.json`.
  - Auto‑titles threads based on the first user message.

- **In‑app cart with PartSelect checkout handoff**
  - Add recommended parts from chat into a cart sidebar.
  - Cart stored per user in `carts.json`.
  - “Checkout on PartSelect” opens real PartSelect product pages in new tabs so the user can add items to the official cart.

- **PartSelect‑style UI**
  - Header, navigation, and hero strip styled after the real site.
  - `/support` in a three‑column layout: threads, chat, and cart.
  - Widgets for install, compatibility, and troubleshooting responses.
  - Markdown rendering for Claude responses.

- **LLM & performance optimisations**
  - Claude Haiku for routing, follow‑up detection, clarification, reranking, summarisation.
  - Claude Sonnet for main answers.
  - History summarisation and response caching to bound latency and cost.

## Tech Stack

- **Frontend**
  - Next.js 16 App Router (React, TypeScript)
  - Tailwind‑style CSS utilities
- **Backend**
  - Python 3.8, FastAPI, Uvicorn
  - Pydantic v2 for schemas
- **LLMs**
  - Anthropic Claude Sonnet (answers)
  - Anthropic Claude Haiku (classification, reranking, summarisation)
- **Embeddings**
  - Voyage (or compatible) embeddings via REST
- **Storage (JSON)**
  - Products, compatibility, repair guides
  - Vector index
  - Users, threads, carts

## Architecture

### High‑level

```text
Browser (Next.js)
  └── /api/chat, /api/auth, /api/threads, /api/cart
        └── FastAPI
             ├── Auth + token parsing
             ├── ThreadStore (threads.json)
             ├── CartStore (carts.json)
             ├── Tools (products, compatibility, guides, vector_store, rerank)
             └── Agents
                  ├── InstallAgent
                  ├── CompatibilityAgent
                  ├── ProductInfoAgent
                  └── TroubleshootingAgent
                       └── Claude (Sonnet + Haiku)
```

- `run_agent` orchestrates:
  - Thread resolution and history loading.
  - Single small classification call (intent + follow‑up + clarification).
  - Optional RAG retrieval + reranking.
  - Agent dispatch and response caching.

## Data Files

All data lives under `backend/app/data`:

- `products.json` – curated product subset with part number, name, URL, price, image.
- `compatibility.json` – model → compatible part numbers.
- `install_guides.jsonl` – one JSON guide per line for installation flows.
- `troubleshooting_guides.jsonl` – guides for common fridge/dishwasher issues.
- `vector_index.json` – embeddings + metadata (generated by `build_index.py`).
- `users.json` – dummy user accounts.
- `threads.json` – thread metadata and messages.
- `carts.json` – per‑user carts.
- `rag_eval_queries.json` – test queries for retrieval evaluation.

## Running Locally

### 1. Backend

```bash
cd backend

# Create and activate venv (if needed)
python -m venv .venv
source .venv/bin/activate        # Linux/macOS
# .venv\Scripts\activate        # Windows

pip install -r requirements.txt

# (Optional but recommended) rebuild vector index
PYTHONPATH=. python scripts/build_index.py

# Start FastAPI
uvicorn app.main:app --reload
```

The backend runs on `http://localhost:8000`.

### 2. Frontend

```bash
cd frontend
npm install
npm run dev
```

The frontend runs on `http://localhost:3000`.

## Usage

1. Visit `http://localhost:3000/login`.
2. Log in as `instalily / instalily` (or `harshit / harshit`).
3. You’ll be redirected to `/support`, which shows:
   - Threads on the left
   - Chat in the middle
   - Cart on the right
4. Try these example prompts:
   - “My refrigerator ice maker is not working”
   - “How do I install part PS11752778?”
   - “Is PS11752778 compatible with model WDT780SAEM1?”
5. Add suggested parts to the cart and click “Checkout on PartSelect” to open the real product pages.

## Evaluation and Testing

- Run RAG evaluation:

```bash
cd backend
PYTHONPATH=. python scripts/eval_rag.py
```

This prints Recall@3 over a small labelled query set.

- Manual tests:
  - Auth: login, signup, logout.
  - Threads: create, switch, resume.
  - Cart: add/remove items, checkout links.
  - Chat: each intent, follow‑up questions, out‑of‑scope prompts.

---

## Limitations & Future Work

- JSON storage is not suitable for high‑throughput production; a relational DB + vector DB would be the natural next step.
- Cart and order flows are one‑way handoffs; real integration would involve secure APIs with PartSelect.
- No analytics dashboard yet; logs exist but aren’t fed into metrics.
